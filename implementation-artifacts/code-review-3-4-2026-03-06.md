## 🟡 MEDIUM — À traiter

### 3. Double fetch possible au changement de produit

Deux `useEffect` appellent `loadMovements` : un au changement de `selectedProductId`, un autre au changement de `pagination.page` / filtres. Cela peut provoquer des appels redondants.

**Fichier :** `apps/web/src/app/(app)/movements/page.tsx` L104-116  
**Action :** Fusionner ou simplifier la logique pour éviter les doubles appels (ex. un seul `useEffect` avec toutes les dépendances).

---

### 4. Produit hors limite 200 — pré-sélection depuis Stocks

Navigation depuis Stocks avec `?product_id=xxx`. Si le produit n'est pas dans les 200 premiers de `GET /products?limit=200`, le select affiche une valeur invalide (UUID sans option correspondante).

**Fichier :** `apps/web/src/app/(app)/movements/page.tsx`  
**Action :** Si `product_id` est en URL et absent de la liste, charger ce produit via `GET /products/:id` et l'ajouter à la liste (ou afficher un message explicite).

### 5. Tests pos_sale non couverts

## 🟢 LOW — Améliorations

### 7. Bouton « Filtrer » redondant

Les filtres déclenchent déjà `loadMovements` via `useEffect` (L114-116). Le bouton « Filtrer » est donc redondant.

**Action :** Soit supprimer le bouton, soit le garder pour une action explicite « Actualiser » (cohérent avec la page Stocks).

### 8. Race condition sur loadMovements

En cas de changement rapide de produit, les réponses peuvent arriver dans le désordre. Pas d’annulation de requête en cours.

**Action :** Utiliser `AbortController` dans `fetchApi` ou ignorer les réponses obsolètes via un compteur/flag.

---

### 9. Rétention Premium Plus : doc vs implémentation

L’AC indique « illimité » pour Premium Plus, les Dev Notes parlent de 365 jours. Le service utilise `history_days` de l’abonnement.

**Action :** Aligner la doc (AC / Dev Notes) avec le comportement réel (365 ou illimité).
