## 🟡 MEDIUM

### 1. Double appel à `loadMovements` au chargement initial

**Fichier :** `apps/web/src/app/(app)/movements/page.tsx`

Deux `useEffect` appellent `loadMovements()` lorsque `selectedProductId` et `token` sont définis :

- Lignes 106–115 : au changement de `selectedProductId` ou `token`, on fait `setPagination(..., page: 1)` puis `loadMovements()`.
- Lignes 121–124 : dépendances `[pagination.page, filterType, filterUserId, filterDateFrom, filterDateTo, loadMovements]` ; lorsque `selectedProductId` et `token` sont déjà renseignés (ex. lien depuis Stocks avec `?product_id=...`), cet effet s’exécute aussi au montage.

Résultat : au premier affichage avec produit présélectionné, la liste des mouvements est chargée deux fois (double requête GET, mauvaise UX / charge inutile).

**Action :** Centraliser le chargement dans un seul effet (ex. un seul `useEffect` qui dépend de `selectedProductId`, `token`, `pagination.page` et des filtres) ou utiliser un flag / ref pour éviter le second appel au montage.

---

### 2. Filtre « Utilisateur (UUID) » : 400 non expliqué

**Fichier :** `apps/web/src/app/(app)/movements/page.tsx`

Si l’utilisateur saisit un texte qui n’est pas un UUID valide, l’API renvoie 400 (`user_id` validé par `isUUID()`). Le front affiche le message générique « Impossible de charger les mouvements. », sans indiquer que le filtre utilisateur est invalide.

**Action :** Soit valider le format UUID côté client avant l’appel (et afficher un message du type « Identifiant utilisateur invalide »), soit interpréter la réponse 400 et afficher un message d’erreur plus explicite lorsque le corps ou les headers indiquent une erreur de validation.

---

### 3. Produit présélectionné hors des 200 premiers (pré-sélection depuis Stocks)

**Fichier :** `apps/web/src/app/(app)/movements/page.tsx`

Quand on arrive depuis la page Stocks avec `?product_id=xxx`, le select est alimenté par `GET /products?limit=200`. Si le produit concerné n’est pas dans ces 200 premiers, il n’apparaît pas dans la liste déroulante : l’URL contient un `product_id` valide mais le select n’a pas d’option correspondante (comportement confus).

**Action :** Si `product_id` est présent dans l’URL et absent de la liste `products`, appeler `GET /products/:id` pour ce produit et l’ajouter à la liste (ou à une option « Produit présélectionné »), ou afficher un message explicite (« Ce produit n’est pas dans la liste ») et garder la sélection affichée.

---

### 4. Aucun test automatisé frontend pour la page Mouvements

**Story :** « Test manuel : flux complet implémenté ». Il n’existe pas de test composant (ex. React Testing Library) ni E2E pour la page `/movements` (sélection produit, chargement, filtres, pagination, export CSV). Toute régression passe inaperçue.

**Action :** Ajouter au moins un test smoke (ex. rendu de la page, sélection produit, appel API mocké ou test E2E minimal) pour sécuriser la non-régression.

## 🟢 LOW

### 5. Bouton « Exporter CSV » et AC « désactivé si aucun produit »

**Fichier :** `apps/web/src/app/(app)/movements/page.tsx`

L’AC demande un bouton « désactivé si aucun produit sélectionné ». Aujourd’hui le bouton n’est rendu que lorsque `selectedProductId` est truthy, donc il n’est jamais visible sans produit. Le comportement est correct, mais si un refactor déplace le bouton hors du bloc conditionnel, il pourrait apparaître actif sans produit.

**Action :** Rendre le bouton toujours visible (ex. dans la barre d’outils) avec `disabled={!selectedProductId}` pour être explicite et résilient au refactoring.

---

### 6. Validation de `product_id` dans l’URL

**Fichier :** `apps/web/src/app/(app)/movements/page.tsx`

`product_id` vient de `searchParams.get('product_id')` et est utilisé tel quel. Si l’URL contient un non-UUID, l’API renverra 400 ou 404. Une validation côté client (format UUID) permettrait d’éviter des appels inutiles et d’afficher un message clair (« Identifiant produit invalide »).

**Action :** Valider le format UUID (regex ou lib) avant d’appeler les APIs ou de pré-remplir le select.

---

### 7. Rétention « illimité » vs 365 jours

**Story / Dev Notes :** L’AC mentionne « illimité » pour Premium Plus ; les Dev Notes et le code (subscription `history_days`) utilisent 365 jours. Comportement cohérent côté code, mais la doc utilisateur peut prêter à confusion.

**Action :** Aligner la rédaction (AC / Dev Notes) avec le comportement réel (ex. « 365 jours » ou « jusqu’à 365 jours » pour Premium Plus).
