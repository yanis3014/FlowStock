## MEDIUM

### M1 — Silence pour tenant n’ayant jamais reçu d’événement

**AC :** « Aucun événement reçu pendant une durée configurée (ex. 15 min) » → mode dégradé.

**Constat :** Dans `evaluateDegradedForTenant`, `silenceExceeded = lastEventMs != null && now - lastEventMs > silenceMs`. Si `last_event_received_at` est `NULL` (tenant avec POS actif mais aucun webhook encore reçu), `silenceExceeded` est toujours faux. Un nouveau tenant ne sera donc jamais marqué dégradé pour « silence », même après 15 min sans aucun événement.

**Fichiers :** `apps/api/src/services/pos-sync-status.service.ts` (l. 98–99)

**Recommandation :** Décider du comportement attendu (dégradé après X min sans premier événement ou seulement après au moins un événement). Si le produit exige l’alerte « pas d’événement depuis 15 min » dès la mise en service, traiter le cas `last_event_received_at IS NULL` (ex. utiliser `created_at` de la config ou une date de « mise en attente »).

## LOW

### L1 — Race condition sur `webhook_failure_count`

**Constat :** `recordWebhookFailure` fait `UPDATE tenant_pos_config SET webhook_failure_count = webhook_failure_count + 1 ...` sans `SELECT FOR UPDATE` ni transaction dédiée. En cas d’échecs webhook concurrents (même tenant), des incréments peuvent être « perdus » (lost update). Impact pratique limité (seuil de 5), mais incohérent avec un compteur strictement consécutif.

**Fichiers :** `apps/api/src/services/pos-sync-status.service.ts`

**Recommandation :** Utiliser `UPDATE ... SET webhook_failure_count = webhook_failure_count + 1 ... WHERE tenant_id = $1 RETURNING webhook_failure_count` (déjà le cas) ; si besoin de garantie forte, envelopper dans une transaction avec lock sur la ligne (SELECT FOR UPDATE) avant l’UPDATE.

---

### L2 — Métriques support / vue admin

**Story :** « Exposer les métriques pour le support : dernier événement reçu, nombre d’échecs, statut dégradé. Endpoint dédié ou section admin, protégé par auth tenant/admin. »

**Constat :** GET `/dashboard/pos-sync-status` (auth tenant) expose bien ces champs pour le tenant courant. Il n’y a pas d’endpoint admin (ex. GET `/admin/tenants/:id/pos-sync-status`) pour qu’un support consulte le statut d’un tenant arbitraire. Optionnel selon la story (« ou section admin »).

**Recommandation :** Backlog : ajouter une route admin protégée si le support doit consulter le statut POS de n’importe quel tenant sans se faire passer pour ce tenant.
