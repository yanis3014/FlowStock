## 🔴 CRITICAL / HIGH

### 1. Label d’aperçu faux (prévisualisation)

- **Fichier :** `apps/web/src/app/(app)/sales/page.tsx` (ligne ~541)
- **Constat :** Le libellé indique « Aperçu (20 premières lignes) » alors que le code affiche `importPreview.sampleRows.slice(0, 5)` → **5 lignes seulement**.
- **Impact :** Incohérence entre l’interface et le comportement ; l’AC « une prévisualisation des données est affichée » et la tâche 1.3 (aperçu) sont trompeuses pour l’utilisateur.

## 🟡 MEDIUM

### 2. Aucun toast après fermeture si 0 ventes importées et des erreurs

- **Fichier :** `apps/web/src/app/(app)/sales/page.tsx` — `handleImportCloseAndRefresh` (lignes 348–356)
- **Constat :** Le toast n’est affiché que si `imported > 0`. Si l’import échoue pour toutes les lignes (`imported === 0`, `errors.length > 0`), l’utilisateur ferme le modal sans aucun retour global (toast).
- **Impact :** UX incohérente ; pas de confirmation claire qu’aucune vente n’a été importée et que des erreurs existent.
- **Action :** Afficher un toast dans ce cas, ex. « Aucune vente importée. X erreur(s) rencontrées. »

### 3. Pas de focus trap sur le modal Import CSV

- **Fichier :** `apps/web/src/app/(app)/sales/page.tsx` — modal Import (lignes 485–593)
- **Constat :** Les modals Création/Édition et Suppression utilisent `modalContentRef` / `modalDeleteRef` et `setupFocusTrap`. Le modal Import n’a ni ref ni focus trap.
- **Impact :** Accessibilité et cohérence avec la Story 3.6 (focus trap et Escape dans les modals). Tab peut sortir du modal.
- **Action :** Ajouter un ref sur le contenu du modal Import et appliquer le même `setupFocusTrap` que pour les autres modals.

### 4. File List potentiellement incomplète

- **Story :** Section « File List » — uniquement `page.tsx` listé.
- **Constat :** Git montre des modifications sur d’autres fichiers (README, migrations, package.json, scripts, token.txt, sprint-status). Si certains relèvent de la story 3.7, la File List est incomplète.
- **Action :** Vérifier l’origine des changements et mettre à jour la File List si des fichiers modifiés pour 3.7 ne sont pas listés.

### 5. Pas de garde-fou si product_sku n’est pas mappé

- **Fichier :** `apps/web/src/app/(app)/sales/page.tsx` — étape 2, bouton « Lancer l’import »
- **Constat :** L’utilisateur peut lancer l’import sans mapper aucune colonne à `product_sku`. L’API renverra alors des erreurs pour chaque ligne. Aucune validation côté UI.
- **Impact :** Expérience dégradée : import « réussi » côté requête mais 0 ligne importée et beaucoup d’erreurs, sans explication proactive.
- **Action :** Désactiver « Lancer l’import » ou afficher un avertissement si la colonne Produit (SKU) n’est pas mappée (au moins un mapping vers `product_sku`).

## 🟢 LOW

### 6. Téléchargement du template : fetch brut au lieu de useApi

- **Fichier :** `apps/web/src/app/(app)/sales/page.tsx` — `downloadTemplate` (lignes 355–369)
- **Constat :** Utilisation de `fetch` direct avec `NEXT_PUBLIC_API_URL` et `Authorization`, sans passer par `useApi`. Cohérence et gestion des cookies/credentials pourraient être alignées sur le reste de l’app.
- **Action :** Optionnel : utiliser la même base URL / mécanisme que useApi pour le GET template (ou documenter la raison du fetch direct).

### 7. Erreurs d’import limitées à 20 dans l’UI

- **Fichier :** `apps/web/src/app/(app)/sales/page.tsx` (lignes 563–570)
- **Constat :** Seules les 20 premières erreurs sont listées, avec un texte « … et X autre(s) erreur(s) ». Comportement acceptable mais limité pour les gros imports.
- **Action :** Amélioration possible : lien « Voir tout » ou export des erreurs (ex. CSV) pour les imports avec beaucoup d’erreurs.

### 8. Tests d’intégration non vérifiés

- **Story :** Tâche 3.1 — « tests d’intégration API existants pour l’import restent verts ».
- **Constat :** Exécution des tests impossible dans l’environnement de revue (Postgres non disponible / auth échouée). Le fichier `sales-import.integration.test.ts` existe et couvre template, preview, import, mapping, multi-tenant.
- **Action :** Exécuter `npm run test -- --testPathPattern="sales-import"` en CI ou avec Postgres local et confirmer que tous les tests passent.

### 9. Indentation incohérente dans `loadSales`

- **Fichier :** `apps/web/src/app/(app)/sales/page.tsx` (lignes 85–91)
- **Constat :** Dans le callback de `loadSales`, les lignes `if (dateFrom) params.set(...)` etc. sont indentées avec 2 espaces au lieu d'être alignées avec le bloc (6 espaces). Incohérence de style.
- **Action :** Indenter correctement le bloc pour respecter le style du fichier.

### 10. Champ `ignored` du rapport d'import non affiché

- **Fichier :** `apps/web/src/app/(app)/sales/page.tsx` — état `importResult`, réponse API
- **Constat :** L'API renvoie `{ imported, errors, ignored, totalRows }`. Le front n'affiche pas `ignored` (lignes ignorées). Le rapport à l'étape 3 pourrait être plus complet.
- **Action :** Optionnel : afficher « X ligne(s) ignorée(s) » dans le bloc résultat lorsque `ignored > 0`.
