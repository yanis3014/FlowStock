### MEDIUM

1. **Retry CSRF incompatible avec FormData (useApi)**  
   - **Fichier :** `apps/web/src/hooks/useApi.ts` lignes 74-94  
   - **Problème :** En cas de 403 CSRF, le retry ne s’applique que si `isBodyReplayable(options.body)` est vrai (body de type string). Pour `FormData`, le body n’est pas rejouable, donc pas de retry. Les POST `/products/import/preview` et `/products/import` en FormData peuvent échouer définitivement si le token CSRF a expiré.  
   - **Correction :** Documenter la limitation ; le message « Session expirée. Veuillez réessayer en sélectionnant à nouveau le fichier » est déjà affiché côté import-stocks. Une amélioration serait de pré-récupérer un token CSRF frais avant l’upload si la session est longue.

   - **Problème :** Le bouton dit « Télécharger le template CSV » alors que l’import supporte aussi Excel. Le libellé pourrait préciser « CSV (compatible Excel) » si pertinent.

## Findings Story 3.3

### CRITICAL

*Aucun.*

### HIGH

1. **Liste des emplacements affiche les emplacements soft-deleted**  
   - **Fichier :** `apps/web/src/app/(app)/locations/page.tsx` ligne 49  
   - **Problème :** `GET /locations?limit=100` est appelé sans `is_active=true`. L’API retourne tous les emplacements, y compris ceux supprimés (soft delete, `is_active=false`). Après suppression, l’emplacement reste visible dans la liste, ce qui crée une confusion UX : l’utilisateur pense avoir supprimé mais l’élément est toujours là.  
   - **Référence :** AC 1 « je peux modifier et supprimer des emplacements » — après suppression, l’emplacement ne devrait plus apparaître par défaut.  
   - **Correction :** Passer `is_active=true` en query param : `fetchApi('/locations?limit=100&is_active=true')`.

2. **Édition : pas de rafraîchissement via GET /locations/:id**  
   - **Fichier :** `apps/web/src/app/(app)/locations/page.tsx` lignes 76-85  
   - **Problème :** Task 1.2 : « modal « Modifier » pré-rempli via GET /locations/:id ». L’implémentation pré-remplit à partir des données de la liste (`loc`), pas d’un appel GET frais. En cas de données modifiées ailleurs (autre onglet, autre utilisateur), le formulaire affiche des données obsolètes.  
   - **Correction :** Au clic sur « Modifier », appeler `GET /locations/:id` pour charger les données à jour avant d’ouvrir le modal.

### MEDIUM

3. **Gestion d’erreur confirmDelete incomplète**  
   - **Fichier :** `apps/web/src/app/(app)/locations/page.tsx` lignes 166-169  
   - **Problème :** `res.json().then((j) => setMessageError(...))` — si `res.json()` échoue (réponse non-JSON), la promesse est rejetée sans `.catch()`. Risque d’unhandled rejection.  
   - **Correction :** `.catch(() => setMessageError('Erreur lors de la suppression.'))` ou gérer le rejet.

4. **Fichiers modifiés non documentés dans File List 3.3**  
   - **Problème :** La story 3.3 modifie `stocks/page.tsx` (filtre emplacement). Ce fichier est aussi dans la File List de la story 3.1. La 3.3 le documente. Mais `nav-config.ts` contient `/locations` — si ce lien a été ajouté pour la story 3.3, il n’est pas listé. Vérification : nav-config a déjà `/locations` (ligne 22) ; probablement préexistant. Pas de problème.

### LOW

5. **Type « Emplacement » en texte libre**  
   - **Fichier :** `apps/web/src/app/(app)/locations/page.tsx` lignes 361-368  
   - **Problème :** Le champ « Type » est un input texte libre. La Dev Note mentionne « type (optionnel) » ; l’API accepte toute chaîne. Pour une meilleure cohérence, un select avec des valeurs prédéfinies (ex. Entrepôt, Magasin) pourrait être proposé.  
   - **Note :** Acceptable pour MVP.

6. **Pas de tests frontend**  
   - **Référence :** Task 4.2 « Test manuel des flux création, édition, suppression, filtre emplacement sur Stocks ».  
   - **Problème :** Aucun test automatisé frontend. Les tests API locations existent et sont complets.
