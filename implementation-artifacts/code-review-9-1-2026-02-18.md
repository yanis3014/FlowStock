# Code Review - Story 9.1 : Invisible Auth & Session Management

**Date:** 2026-02-18  
**Story:** 9.1  
**Status:** ✅ **APPROUVÉ avec correctifs mineurs recommandés**

## Résumé exécutif

L’implémentation de la Story 9.1 est **conforme aux critères d’acceptation**. L’authentification est transparente (aucun champ JWT visible), un `AuthProvider` et un hook `useApi` centralisent le token et les appels API, et le flux 401/403 → redirection login avec message « Session expirée » est en place. Un correctif est recommandé sur la gestion de `isLoading` lorsque `/auth/me` échoue.

### Points forts ✅

1. **Aucun champ JWT visible** : login, register et dashboard n’affichent aucun input/textarea pour token (vérification grep dans `apps/web/src`).
2. **AuthProvider** : état `user` / `token` / `isLoading`, `login` / `logout` / `setToken`, token en **sessionStorage** (`bmad_jwt_token`), `useAuth()` exposé et utilisé correctement.
3. **useApi** : lecture du token depuis le contexte, CSRF (GET `/csrf-token`, header `X-CSRF-Token` sur POST/PUT/PATCH/DELETE), `credentials: 'include'`, retry CSRF sur 403 avec message "CSRF", intercepteur 401/403 → `setToken(null)` + redirection `/login?session_expired=1` (hors routes `auth/`).
4. **Pages login / register** : formulaires email + mot de passe (et champs inscription sur register) uniquement ; pas de champ token ; message « Session expirée » affiché lorsque `session_expired=1` sur la page login.
5. **Dashboard protégé** : utilise `useApi` pour `GET /dashboard/summary`, redirection vers `/login` si pas de token, aucun champ JWT.
6. **Cohérence avec api-client.js** : logique CSRF, 401/403, retry CSRF et timeout reproduite de façon fidèle dans `useApi`.
7. **Layout** : `AuthProvider` enveloppe l’application dans `layout.tsx`.
8. **Design** : Tailwind avec couleurs `primary`, `success`, `warning`, `error` dans `tailwind.config.ts`.

### 3. Endpoint /auth/me

L’API expose bien `GET /auth/me` (authenticateToken) et renvoie `{ success: true, data: { userId, tenantId, role, email } }`. L’AuthContext l’utilise au chargement pour hydrater `user` à partir du token en sessionStorage.

### 4. Détection des routes auth (401/403)

Dans `useApi.ts`, `isAuthRoute(path)` normalise le path (sans `?`) et vérifie `auth/` en préfixe. Les URLs passées à `fetchApi` sont du type `/dashboard/summary` ; le path utilisé pour la redirection est correct, les routes sous `/auth/*` ne déclenchent pas la redirection.

## Points à corriger ou à améliorer

### 1. Bug : `isLoading` jamais mis à `false` si `/auth/me` répond 4xx

**Impact:** 🟠 **Moyen** (bloquant UX)

Dans `AuthContext.tsx`, lorsque le token existe et que `GET /auth/me` est appelé, si la réponse est `!res.ok` (ex. 401), le code fait `setToken(null)` et `return` sans jamais mettre `isLoading` à `false`. L’utilisateur reste sur un écran « Chargement... » indéfiniment.

**Extrait actuel (lignes 113–119) :**

```ts
.then((res) => {
  if (!res.ok) {
    setToken(null);
    return;  // isLoading reste true
  }
  return res.json();
})
```

**Recommandation :** dans le premier `.then`, en cas de `!res.ok`, mettre aussi `isLoading` à `false` avant de quitter, par exemple :

```ts
.then((res) => {
  if (!res.ok) {
    setToken(null);
    setState((prev) => ({ ...prev, isLoading: false }));
    return;
  }
  return res.json();
})
```

### 2. Register : appel API hors client centralisé

**Impact:** 🟡 **Mineur**

La page register utilise un `fetch` direct (CSRF + `/auth/register`) au lieu d’un client partagé. Pour la story 9.1 c’est acceptable (pas de token avant inscription). Pour homogénéité et évolution (retry CSRF, timeout), on pourrait introduire plus tard un `fetchWithoutAuth()` ou étendre `useApi` pour des appels sans token.

### 3. Configuration API et proxy

**Impact:** 🟡 **Mineur**

- `.env.local.example` indique `NEXT_PUBLIC_API_URL=http://localhost:3000`. En pratique, Next tourne souvent sur 3000 et l’API Express sur un autre port (ex. 3001). À clarifier en doc ou dans le README (quelle URL selon l’environnement).
- `next.config.mjs` n’inclut pas de proxy vers l’API ; les appels utilisent `NEXT_PUBLIC_API_URL`. C’est cohérent avec l’implémentation actuelle ; en dev, CORS doit être configuré côté API pour l’origine du front Next.

### 4. Page d’accueil (`/`) et protection des routes

**Impact:** 🟡 **Mineur**

La page d’accueil est publique (liens Connexion / Créer un compte). La story 9.1 indique comme optionnel un route guard / middleware pour rediriger les non-authentifiés vers `/login`. Ce n’est pas exigé pour 9.1 ; à considérer pour les stories suivantes (routes protégées globales).

## Synthèse

- **Fonctionnel :** Les critères d’acceptation de la story 9.1 sont remplis (auth invisible, session en sessionStorage, useApi, login/register/dashboard sans champ JWT, 401/403 → login + « Session expirée »).
- **Correctif recommandé avant merge :** mettre `isLoading` à `false` dans `AuthContext` lorsque `/auth/me` répond avec `!res.ok`.
- **Recommandations optionnelles :** documenter `NEXT_PUBLIC_API_URL` et les ports en dev ; à terme, centraliser l’appel register (ou appels sans token) si besoin de cohérence avec useApi.
