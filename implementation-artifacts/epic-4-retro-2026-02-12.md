## Succès et Points Forts

### Architecture et stack

- **Chat IA dans le ML Service :** Décision claire (Python/FastAPI) pour LLM et partage d’infra ML ; intégration avec l’API Stocks (Node.js) via appels HTTP internes.
- **Cohérence frontend :** Pages HTML dans `apps/api/public/` (chat, dashboard, forecast, stats) avec `api-client.js`, JWT, design responsive commun.
- **Dashboard comme hub :** GET `/dashboard/summary` centralise ventes veille, stock, alertes, actions recommandées ; réutilisé par les stories 4.4 et 4.5.
- **Multi-tenant :** RLS PostgreSQL (migration V012 pour chat), JWT tenant_id/user_id côté API et ML service.

### Qualité et livrables

- **4.1 – Chat :** 44 tests Python (chat_service, auth, routes), migration V012 idempotente, doc OpenAPI section 12. Mémoire contextuelle (10 derniers messages + chat_context), intégration Stocks (produits, stock-estimates).
- **4.2 – Dashboard :** Tests E2E Playwright (workflow complet, couleurs, responsive, perf < 2s), cache 5s, chargement parallèle, WCAG AA.
- **4.3 – Courbes :** Réutilisation GET `/stock-estimates`, courbes par produit, comparaison multi-produits, marqueur « recommander commande », confidence_level.
- **4.4 – Alertes :** Filtres type/sévérité/statut, marquer vue/résolue (localStorage), seuil configurable (GET/PUT `/dashboard/alert-threshold`).
- **4.5 – Stats :** Page stats.html, ventes hier + stock via dashboard summary, graphique 7j/30j (Chart.js), top produits, export CSV, E2E stats.spec.ts.

### Patterns récurrents

- Réutilisation des services existants (dashboard.service, sales.service, stock-estimates) plutôt que duplication.
- Codes couleur cohérents (vert/orange/rouge) et contrastes WCAG AA.
- Gestion loading/error et états vides sur toutes les pages.

## Défis et Axes d'Amélioration

### Problèmes identifiés (code review 4.4 et analyse)

1. **TypeScript dans HTML (4.4) – CRITIQUE**  
   `dashboard.html` contient `as HTMLInputElement` dans un script exécuté en JS pur → erreur de syntaxe possible. **Action :** Supprimer le cast ou ne garder que du JS pur dans les scripts inline.

2. **Logique « rupture imminente » incorrecte (4.4)**  
   Dans `dashboard.service.ts`, `estimatedDaysRemaining = product.quantity / (product.quantity / 30)` vaut toujours 30. **Action :** Utiliser une vraie estimation (ex. consommation moyenne / stock-estimates) ou désactiver jusqu’à intégration IA.

3. **AC notifications optionnelles (4.4) – PARTIEL**  
   Préférence `bmad_notifications_enabled` en localStorage mais **aucune UI** pour activer/désactiver les notifications. **Action :** Ajouter une case à cocher (header ou section Alertes).

4. **Liens depuis les alertes (4.4)**  
   Pas de lien « Voir le produit » / « Voir la prévision » depuis une alerte. **Action :** Ajouter lien vers fiche produit ou page prévision.

5. **Filtres alertes non persistés (4.4)**  
   Filtres en mémoire uniquement ; rechargement réinitialise. **Action :** Persister en sessionStorage ou query params.

6. **Tests 4.4**  
   Pas de tests d’intégration pour GET/PUT `/dashboard/alert-threshold` ; pas d’E2E pour filtres, marquer résolue, seuil, préférence notifications. **Action :** Ajouter au moins un scénario E2E et tests intégration alert-threshold.

7. **Documentation Dev Agent Record (4.4)**  
   File List vide dans la story 4.4. **Action :** Compléter avec fichiers modifiés et résumé.

8. **Tests E2E chat (4.1)**  
   Non implémentés (pas de framework E2E côté ML/frontend chat). À planifier si besoin de couverture E2E du flux chat.

### Complexité technique

- **4.1 :** Intégration multi-service (ML ↔ API Stocks), JWT partagé, mémoire contextuelle et construction du prompt LLM ; bien décomposé en tâches et testé.
- **4.2–4.5 :** Montée en charge du dashboard (résumé, alertes, courbes, stats) ; garder un œil sur les requêtes et le cache (NFR1 < 2s).
