## Défis et Axes d'Amélioration

### Complexité Technique

- **Moteur de formules personnalisées (Story 3.4) :** Complexité élevée avec mathjs, résolution de variables, validation syntaxe. Bonne séparation mais nécessite une bonne compréhension.
- **Optimisation batch (Story 3.5) :** Requiert une bonne compréhension SQL pour éviter N+1. Pattern bien documenté pour réutilisation.

### Gestion des Erreurs

- **Import CSV (Story 3.2) :** Gestion d'erreurs par lot bien implémentée, mais pourrait bénéficier d'un retry automatique pour certaines erreurs transitoires (futur).
- **Validation formules (Story 3.4) :** Messages d'erreur clairs, mais pourrait avoir une validation plus granulaire (suggestion d'amélioration).

### Documentation

- **Formules prédéfinies :** Documentation disponible dans l'interface, mais pourrait être enrichie avec des exemples concrets d'utilisation.
- **API :** Endpoints documentés, mais OpenAPI/Swagger pourrait être plus complet (amélioration future).

## Apprentissages Clés

### Techniques

1. **Requêtes batch optimisées :** Utiliser GROUP BY au lieu de boucles N+1 pour les agrégations multi-produits
2. **Sandboxing mathjs :** Configuration sécurisée du moteur de formules pour éviter les injections
3. **Validation progressive :** Validation syntaxe → validation variables → évaluation pour meilleure UX
4. **Tri en mémoire vs DB :** Pour petites collections (< 1000), tri en mémoire après agrégation est approprié

### Processus

1. **Réutilisation de services :** Créer des services réutilisables (sales.service.ts) facilite l'implémentation de nouvelles fonctionnalités
2. **Tests d'intégration :** Essentiels pour vérifier l'isolation multi-tenant et les flux complets
3. **Code review systématique :** Identifie les patterns récurrents et améliore la qualité globale

### Architecture

1. **Séparation des responsabilités :** Services distincts (formula vs stock-estimate) évitent le couplage
2. **Isolation multi-tenant :** Toutes les requêtes via queryWithTenant + filtres explicites garantissent la sécurité
3. **Performance :** Optimisations batch critiques pour répondre aux NFR (< 2s réponse)
