## HIGH

### H1. Erreurs 5xx non loguées avec tenant_id (AC non respecté)

- **Fichier:** `apps/api/src/routes/webhooks.pos.routes.ts`
- **AC :** « les erreurs 4xx/5xx sont loguées avec tenant_id pour diagnostic ».
- **Constat :** La route n’a pas de `try/catch`. Si `getConfigByTenantAndSecret`, `validatePayload` ou `recordEventIdempotent` lance une exception (ex. DB indisponible), Express renvoie 500 sans appeler `logWebhookError`. Les 5xx ne sont donc pas logués avec `tenant_id`.
- **Action :** Envelopper le handler dans un `try/catch`, logger via `logWebhookError(500, tenantId, externalId, 'server_error', message)` et renvoyer 500 avec un body générique.

### H2. Validation du format UUID pour X-Tenant-Id

- **Fichiers:** `apps/api/src/routes/webhooks.pos.routes.ts`, service si besoin.
- **Constat :** `tenantId` est passé tel quel à `getConfigByTenantAndSecret`. Si le client envoie `X-Tenant-Id: not-a-uuid`, la requête PostgreSQL peut lever une exception (invalid input syntax for type uuid), ce qui retombe dans le cas 5xx non logué (H1). De plus, accepter une valeur non-UUID complique le diagnostic.
- **Action :** Valider que `tenantId` est un UUID valide avant tout appel DB (regex ou lib). Si invalide → 400 ou 401 avec message clair et log 4xx avec tenant_id si possible (ou valeur reçue tronquée).
