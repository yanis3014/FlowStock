## 4. HIGH

### H1 — Tâche 2.3 Square : vérification de signature non implémentée

- **Où :** Story Task 2.3 — « Vérifier la signature Square si documentée (X-Signature ou header dédié). Répondre 2xx rapidement. »
- **Constat :** Square documente la vérification de signature via le header **X-Square-Signature** (HMAC-SHA256 du body brut, base64). Le code n’effectue aucune vérification : toute requête POST avec un Bearer valide est traitée.
- **Risque :** Usurpation de webhook (replay ou envoi forgé si le secret Bearer fuit ou est deviné). La signature Square est la garantie que le body provient bien de Square.
- **Preuve :** `apps/api/src/routes/webhooks.pos.routes.ts` — route `/square` : pas de lecture de `X-Square-Signature`, pas d’appel à un vérificateur. Réf. [Square — Verify and Validate an Event Notification](https://developer.squareup.com/docs/webhooks/step3validate).
- **Recommandation :** Implémenter la vérification optionnelle : si le tenant a une « signature key » Square configurée (ex. champ `square_signature_key` dans `tenant_pos_config` ou table dédiée), calculer HMAC-SHA256 du body brut, comparer au header `X-Square-Signature`, et renvoyer 401 si présent mais invalide. Sinon, garder le comportement actuel pour rétrocompatibilité.

## 5. MEDIUM

### M1 — OpenAPI : schéma 200 détaillé manquant pour /laddition et /square

- **Où :** `apps/api/src/openapi/spec.ts` — paths `/webhooks/pos/laddition`, `/webhooks/pos/square`.
- **Constat :** La description indique « même format que POST /webhooks/pos », mais le path générique `/webhooks/pos` ne décrit pas non plus dans son schéma 200 les champs `processed_lines`, `unmapped_lines`, `errors`, `details`. Seul le texte de description les mentionne. Les consommateurs de l’OpenAPI ne peuvent pas s’appuyer sur un schéma précis pour ces réponses.
- **Recommandation :** Introduire un schéma réutilisable pour la réponse 200 (received?, duplicate?, processed_lines, unmapped_lines, errors, details?) et l’utiliser pour `/webhooks/pos`, `/webhooks/pos/lightspeed`, `/webhooks/pos/laddition`, `/webhooks/pos/square`.

### M2 — Duplication de code dans les routes POS

- **Où :** `apps/api/src/routes/webhooks.pos.routes.ts` — handlers `/lightspeed`, `/laddition`, `/square`.
- **Constat :** Chaque handler répète ~50 lignes : lecture tenant/secret, validation UUID, `getConfigByTenantAndSecret`, vérification `pos_type`, parsing body (form vs JSON), parse adapter, transform, `recordEventIdempotent`, `processPosSaleEvent`, réponse 200. Toute évolution (ex. ajout de métriques, correction de log) doit être répliquée trois fois.
- **Recommandation :** Extraire une fonction helper du type `handlePosWebhook(req, res, { posType, parsePayload, transformToInternal })` et faire appel à celle-ci depuis les trois routes. Réduit les oublis (ex. externalId dans les catch).

### M3 — Logging 500 : externalId manquant pour /laddition et /square

- **Où :** `webhooks.pos.routes.ts` — blocs `catch` des routes `/laddition` (l.258–261) et `/square` (l.318–321).
- **Constat :** On appelle `logWebhookError(500, tenantId, null, 'server_error', message)` avec `externalId = null`. Pour Lightspeed, le code tente d’extraire un identifiant du body (saleID ou payload) pour le log. En cas d’erreur 500 après parse réussi, avoir l’external_id dans les logs facilite le debug.
- **Recommandation :** Dans le catch de `/laddition`, passer `sale?.saleId ?? null` (ou équivalent depuis req.body si sale non défini). Pour `/square`, passer `order?.orderId ?? null` ou extraire depuis `req.body` (event_id / data.object.order.id). Aligner sur le pattern Lightspeed.

### M4 — Test manquant : 403 lorsque pos_type !== 'square'

- **Où :** `apps/api/src/__tests__/webhooks/pos-webhook.integration.test.ts` — describe `POST /webhooks/pos/square`.
- **Constat :** Le describe L’Addition contient « should return 403 when pos_type is not laddition ». Aucun test équivalent pour Square : un tenant configuré en `pos_type=lightspeed` ou `laddition` qui envoie une requête à `/webhooks/pos/square` devrait recevoir 403.
- **Recommandation :** Ajouter un test du type « should return 403 when pos_type is not square », en mettant temporairement `pos_type` à `lightspeed`, en appelant POST /webhooks/pos/square avec le même secret, puis en restaurant `pos_type=square`.

## 6. LOW

### L1 — Square : pas de support form-urlencoded avec champ "payload"

- **Où :** Route `/square` — elle utilise uniquement `req.body` comme objet ; pas de déballage de `body.payload` (JSON string) comme pour Lightspeed et L’Addition.
- **Constat :** Si un intégrateur ou une passerelle envoie le body Square en `application/x-www-form-urlencoded` avec un champ `payload` contenant le JSON (comme pour Lightspeed/L’Addition), le parser Square recevra `{ payload: "..." }`, ne trouvera pas `data.object.order`, et renverra 400.
- **Impact :** Faible si la doc impose le JSON brut ; toutefois incohérent avec les deux autres connecteurs.
- **Recommandation :** Documenter clairement dans l’OpenAPI que Square n’accepte que `application/json` brut ; ou ajouter le même traitement que pour L’Addition (si `body.payload` est une string, la parser en JSON puis passer à `parseSquarePayload`).

### L2 — Test form-urlencoded manquant pour L’Addition

- **Où :** `pos-webhook.integration.test.ts` — describe L’Addition.
- **Constat :** Lightspeed a un test « should accept form-urlencoded body with payload field ». L’adapter L’Addition gère le même cas (payload dans un champ), mais il n’y a pas de test dédié.
- **Recommandation :** Ajouter un test similaire pour L’Addition : body en form, champ `payload` = JSON string d’un sale valide, attendre 200 et comportement cohérent (unmapped ou processed selon mapping).

### L3 — Square adapter : structure order_updated non couverte par les tests

- **Où :** `square.adapter.ts` — branche `data.object.order_updated` (l.44–51).
- **Constat :** Le code gère une variante « order_updated » pour extraire `order_id` et `created_at`. Les tests n’envoient que `data.object.order`. Les payloads réels Square pour `order.updated` peuvent avoir une structure différente (ex. `order_updated` avec un sous-objet).
- **Recommandation :** Soit ajouter un test avec un payload type `order.updated` (structure documentée Square), soit documenter dans les Dev Notes que seul `order.created` (ou une forme précise) est testé et supporté officiellement.

### L4 — Documentation des limites (réponse 2xx, idempotence)

- **Où :** Story Task 5.1 — « Documenter limites (réponse 2xx, idempotence via external_id). »
- **Constat :** L’OpenAPI décrit bien « Réponse 2xx rapidement » et l’idempotence via external_id. Il manque une phrase explicite du type : « En cas d’échec partiel (lignes non mappées ou stock insuffisant), l’API renvoie malgré tout 200 avec processed_lines / unmapped_lines / errors pour éviter les retries massifs côté POS. »
- **Recommandation :** Ajouter cette précision dans la description des paths L’Addition et Square (et éventuellement Lightspeed) dans spec.ts.
