# Story 8.5: Nettoyage du code — Suppression des balises \<style\> internes

**Status:** Done

## Story

**As a** développeur,  
**I want** qu'aucun fichier .html ne contienne de balises \<style\> internes,  
**so that** l'architecture est propre et centralisée (styles gérés par Tailwind et éventuellement feuilles partagées).

## Acceptance Criteria

**Given** tous les fichiers .html de l'application (apps/api/public ou équivalent)  
**When** on audite le code  
**Then** il n'existe plus de balises `<style>` à l'intérieur des fichiers .html  
**And** les styles auparavant définis dans ces balises ont été migrés vers des classes Tailwind ou des feuilles CSS centralisées (si exception documentée)  
**And** le rendu visuel des pages concernées reste inchangé ou amélioré après la migration

## Tasks / Subtasks

- [x] Task 1 — Audit des balises \<style\>
  - Parcourir tous les .html sous apps/api/public (et autres dossiers front concernés)
  - Lister chaque fichier contenant une ou plusieurs balises `<style>` et le contenu associé (sélecteurs, propriétés)
  - Prioriser les pages métier (dashboard, stats, chat, formules, import, movements, référentiels)

- [x] Task 2 — Migration vers Tailwind ou CSS centralisé
  - Pour chaque bloc `<style>` : soit le remplacer par des classes Tailwind équivalentes sur les éléments concernés, soit déplacer les règles dans une feuille CSS partagée (ex. app.css) si nécessaire (animations, cas complexes)
  - Supprimer la balise `<style>` une fois la migration validée visuellement
  - Documenter toute exception (fichier + raison) si une feuille centralisée est utilisée

- [x] Task 3 — Validation et régression
  - Vérifier chaque page modifiée : rendu identique ou amélioré, pas de régression sur desktop/mobile
  - Confirmer qu’aucune recherche grep/codebase ne trouve plus de `<style>` dans les .html (sauf exception documentée)
