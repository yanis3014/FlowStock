# Story 4.3: Visualisation Interactive des Courbes de Prévision

**Status:** Done

## Story

**As a** gérant de PME,  
**I want** voir des courbes de prévision montrant quand mes stocks vont tomber,  
**so that** je peux comprendre visuellement les tendances et prévisions.

## Acceptance Criteria

**Given** je suis un utilisateur authentifié avec des données de ventes  
**When** je consulte les courbes de prévision  
**Then** les courbes montrent quand les stocks vont tomber  
**And** les courbes montrent quand recommander une commande  
**And** le pourcentage de certitude est affiché à chaque fois (quand IA disponible)  
**And** je peux comparer des produits via bouton dédié  
**And** la visualisation est intuitive et interactive  
**And** les courbes sont affichées dans l'interface (dashboard, vue produit)

## Dev Notes

### Previous Story Insights

- Story 4.2 a livré : Dashboard principal avec vue d'ensemble, statistiques, alertes, actions recommandées. Les courbes peuvent être accessibles depuis le dashboard (lien ou section).
- Story 3.5 a livré : GET `/stock-estimates` et GET `/stock-estimates/:productId` avec current_stock, avg_daily_consumption, days_remaining, estimated_stockout_date, confidence_level. Page stock-estimates.html existe (tableau) ; à enrichir ou compléter par une vue graphique.
- API Predictions (Premium+) : GET `/predictions/:product_id` expose prediction_timeline (série date / predicted_quantity) pour courbes IA [Source: docs/api-specifications.md#4-predictions-service]. En MVP, utiliser stock-estimates ; prévoir extension pour afficher prediction_timeline quand disponible.

### Data Models

- **StockEstimate** (API actuelle) : [Source: implementation-artifacts/3-5-calculs-basiques-temps-stock.md, docs/api-specifications]  
  - product_id, product_name, sku, current_stock, unit, avg_daily_consumption, days_remaining, estimated_stockout_date, confidence_level, sales_days_count, period_days.
- **Product** (pour lead_time_days) : GET `/products/:id` retourne lead_time_days pour calcul "recommander commande avant".
- **Prediction** (Premium+, optionnel) : GET `/predictions/:product_id` retourne prediction_timeline[], confidence_score, predicted_stockout_date [Source: docs/api-specifications.md#4-predictions-service].

### API Specifications

- **GET /stock-estimates** : [Source: docs/api-specifications.md, stock-estimate.routes]  
  - Query : period_days (optionnel). Réponse 200 avec data[] (estimations triées par urgence). Auth JWT requise.
- **GET /stock-estimates/:productId** : Réponse 200 avec estimation pour un produit. Auth JWT requise.
- **GET /products/:id** : Détail produit inclut lead_time_days. Utiliser pour date "recommander commande".
- **GET /predictions/:product_id** (Premium+) : prediction_timeline (série quotidienne), confidence_score. À utiliser si disponible pour courbe IA et pourcentage certitude.

### Component Specifications

- **Structure** : Suivre pattern existant (pages HTML dans apps/api/public ou app React si applicable). Page ou section dédiée aux courbes (ex. forecast.html ou onglet dans stock-estimates.html).
- **Graphique** : Librairie légère côté client (Chart.js, Lightweight Charts, ou SVG). Pas de dépendance lourde ; préférer bundle size raisonnable.
- **Design System** : [Source: planning-artifacts/ux-design-specification.md] — couleurs cohérentes (Navy Blue, Green Electric), typographie Inter, responsive.

### File Locations

- **Frontend** : apps/api/public/ (pages HTML) — stock-estimates.html existant ; possible nouvelle page forecast.html ou courbes intégrées.
- **API** : Aucune nouvelle route backend requise pour MVP (stock-estimates et products déjà disponibles). Optionnel : endpoint agrégé "timeline" si besoin (sinon construire timeline côté client à partir de stock-estimates + product.lead_time_days).
- **Tests** : Tests manuels ou E2E pour "voir courbe", "comparer produits", "voir date recommandation" ; tests unitaires pour calcul timeline si logique côté client.

### Technical Constraints

- **Performance** : Éviter de charger tous les produits pour une seule vue ; charger par page ou par sélection. Courbe par produit : une requête GET /stock-estimates/:id + GET /products/:id suffit.
- **Accessibilité** : Labels pour lecteurs d'écran, contraste courbes/légendes (WCAG AA). Résumé textuel des prévisions (date rupture, date recommandation) en plus du graphique.
- **Compatibilité** : Fonctionner sans API Predictions (MVP = courbes basées sur stock-estimates uniquement) ; lorsque Predictions est disponible, afficher en plus confidence_score et optionnellement prediction_timeline.

### UX Requirements

- **Intuitif** : [Source: docs/prd.md FR14] — Courbes de prévision montrant quand les stocks vont tomber et quand recommander une commande.
- **Comparaison** : Pouvoir comparer plusieurs produits sur un même graphique (bouton dédié).
- **Certitude** : Pourcentage de certitude affiché quand IA disponible ; sinon niveau de confiance (high/medium/low/insufficient) depuis stock-estimates.

## Testing

- **Fonctionnel** : Utilisateur authentifié voit une courbe par produit (stock → rupture) ; date "recommander commande" visible ; comparaison 2+ produits affiche plusieurs courbes.
- **Edge** : Produit sans ventes (days_remaining null) → message "Données insuffisantes" ou pas de courbe.
- **Responsive** : Graphique lisible sur mobile (redimensionnement, légende lisible).
- **Accessibilité** : Résumé textuel et contraste vérifiés.

### Code Quality Assessment

L’implémentation couvre tous les critères d’acceptation de la story 4.3. La page `forecast.html` utilise Chart.js pour les courbes de prévision, réutilise GET `/stock-estimates` et GET `/products`, construit la timeline côté client (aujourd’hui → rupture avec décroissance linéaire), affiche la date « Recommander commande avant » (ligne verticale orange avec plugin Chart.js), les badges de confiance (high/medium/low/insufficient), la comparaison multi-produits avec légende et liste « Retirer », et gère les états vides.
